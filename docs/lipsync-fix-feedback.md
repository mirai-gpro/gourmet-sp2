## 1. 確認事項への回答

### Q1: 150フレーム（5秒分）一括受信しても問題ないか？

**問題ありません。**

`queueLiveExpressionFrames()` は `this.frameBuffer.concat(frames)` で追加するだけです（`LAMAvatar.astro:823`）。
1チャンクあたりのフレーム数に制限はなく、メモリも 150f × 52ch × 8bytes ≒ 60KB 程度で無視できます。

### Q2: 表情フレームの再生は受信直後から順次再生か？

**はい、順次再生です。**

`performance.now() - liveStreamStartTime` の経過時間でフレームインデックスを算出し、
対応するフレームを即座に適用します（`LAMAvatar.astro:612-613`）。
「溜めてから一括再生」ではありません。

### Q3: `interrupted` 時にキュー内の未再生フレームはクリアするか？

**はい、クリアしています。**

`interrupted` イベント → `stopLiveStream()` 呼び出し（`core-controller.ts:146`）
→ `frameBuffer = []`, `liveStreamStartTime = null` でバッファ・タイマーともにリセット（`LAMAvatar.astro:847-855`）。

## 2. 提案 #7（チャンクサイズ変更）への意見

### 結論: 賛成。ただし段階的チャンクサイズを逆提案

全チャンク一律 5秒ではなく、**段階的にサイズを拡大する方式**を提案します。

```
chunk#1: 0.25秒（即発火）  ← 現状通り
chunk#2: 2.0秒
chunk#3〜: 5.0秒
turn_complete: 残りバッファ
```

```python
# relay.py
A2E_FIRST_CHUNK  = 24000 * 2 // 4      #  12,000 bytes = 0.25秒
A2E_SECOND_CHUNK = 24000 * 2 * 2       #  96,000 bytes = 2.0秒
A2E_NORMAL_CHUNK = 24000 * 2 * 5       # 240,000 bytes = 5.0秒
```

### 理由

#### 懸念: 5秒チャンクの「追いつき問題」

フロントエンドは `audioStartTime`（音声再生開始時刻）を基準に expression を再生します。
チャンクが大きいほど、到着遅延がそのまま「ホールド」（最後のフレームで口が止まる）に直結します。

```
一律5秒チャンクの場合:
0.00s  音声開始
0.25s  chunk#1 到着（7f）→ 再生開始
0.50s  frameIndex=15 → バッファ末端(7f)到達 → ホールド開始 ★
5.25s  chunk#2 到着（150f）→ ホールド解除
       → 0.5〜5.25秒の約4.8秒間、口が同じ位置で固まる
```

```
段階的チャンク（0.25s → 2s → 5s）の場合:
0.00s  音声開始
0.25s  chunk#1 到着（7f）→ 再生開始
0.50s  frameIndex=15 → バッファ末端(7f)到達 → ホールド開始 ★
2.25s  chunk#2 到着（60f）→ ホールド解除
       → ホールド時間 = 約1.8秒（許容範囲）
7.25s  chunk#3 到着（150f）→ この時点で十分なバッファがある
```

ユーザーが最も注目する**発話冒頭 2.25秒間**で 2チャンク分のデータが届くため、
冒頭の口の動きの品質が向上します。

#### API コール数の比較（10秒の応答）

| 方式 | タイムライン | 回数 |
|------|------------|------|
| 現在（1秒固定） | 0.25 + 1×10 | **11回** |
| 一律 3秒 | 0.25 + 3 + 3 + 3.75 | **4回** |
| **0.25/2/5 段階** | **0.25 + 2 + 5 + 2.75** | **4回** |
| 一律 5秒 | 0.25 + 5 + 4.75 | **3回** |

**0.25/2/5 段階方式は一律3秒と同じ4回**だが、冒頭の品質が高い。

#### フロントエンド側の変更

**不要。** チャンクサイズが可変でも `queueLiveExpressionFrames()` は
受信フレームを `concat` するだけなので影響なし。

## 3. 追加要望

### 要望1: `turn_complete` 後のゼロチャンク送信を停止（P1）

**現象**: ターン間で 4フレームの全ゼロ expression が到着し、偽のストリームが開始される。

```
フロントエンドのログ:
Turn 1 終了
→ 4フレーム(全ゼロ)到着 → liveStreamStartTime が再セット → 偽ストリーム開始
→ 2秒タイムアウト待ち → 自動終了
→ Turn 2 の expression 受信開始（本来はここからストリーム開始すべき）
```

**影響**: 偽ストリームの 2秒タイムアウト中に Turn 2 の expression が届くと、
偽ストリームのタイムラインに組み込まれてしまい、同期がずれる。

**依頼**: `turn_complete` 送信後は expression メッセージを送信しないようにしてほしい。

```python
# relay.py 修正案
if sc.turn_complete:
    self._turn_active = False

async def _send_expression_chunk(self, chunk, **kwargs):
    if not self._turn_active:
        return  # turn_complete 後は送信しない
    # ... 通常処理 ...
```

### 要望2: expression メッセージに `chunk_index` / `is_final` を追加（P2）

現在、フロントエンドは「2秒間新チャンクなし」でストリーム終了を判定しています（`LAMAvatar.astro:645`）。
バックエンドが明示的に最終チャンクを通知してくれると、以下が改善されます。

- ストリーム終了判定が正確になる（2秒待たなくてよい）
- バッファ末端到達時、次のチャンクが来るかどうか判断できる
- デバッグ時にチャンクの到着順序を追跡しやすい

**依頼フォーマット**:

```json
{
  "type": "expression",
  "data": {
    "names": ["browInnerUp", "..."],
    "frames": [[0.12, "..."], ["..."]],
    "frame_rate": 30,
    "chunk_index": 0,
    "is_final": false
  }
}
```

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `chunk_index` | `number` | 0始まりのチャンク連番（ターンごとにリセット） |
| `is_final` | `boolean` | このターンの最終チャンクかどうか |

**フロントエンド側の対応**: `is_final: true` を受信したら、バッファ末端到達時に
2秒タイムアウトを待たずにストリームを終了します。こちらで実装します。

### 要望3: jawOpen の出力スケール確認（P3・低優先度）

対策 #5 で全ゼロ問題は解消されましたが、jawOpen の実測値が低い状況です。

```
jawOpen の実測分布:
  0.00-0.05: ████████████████████████████  78%  ← ほぼ閉口
  0.05-0.10: ████                           11%
  0.10-0.20: ███                             8%
  0.20-0.30: █                               3%
  平均: ~0.03  最大: 0.287
```

jawOpen=0.03 は視覚的にほぼ口が閉じた状態で、音声が聞こえても口が動いて見えません。
理想的には **平均 0.08〜0.12、最大 0.4〜0.5** 程度あると自然に見えます。

**依頼**: audio2exp の出力パラメータで jawOpen のスケール調整が可能かどうか確認してほしい。

- 調整可能 → 1.5〜2.0 倍にスケーリング
- 調整不可 → 情報を共有いただければフロントエンドで増幅処理を実装します
